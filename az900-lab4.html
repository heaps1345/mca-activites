    Lab Overview

    Identity is the new security perimeter in cloud computing. In traditional networks, a firewall protected the boundary. In cloud environments, the identity of the user or service is the boundary. Microsoft Entra ID is Azure's cloud identity service - it handles authentication (who are you?) and authorisation (what can you do?) for every resource in your subscription.

    This lab covers the four core identity pillars tested in AZ-900: Entra ID users, RBAC role assignments, Conditional Access policies, and the Zero Trust model. All four feed directly into AZ-500.

    i This lab is entirely free - no compute resources are deployed. Entra ID user management and RBAC assignment are included in every Azure subscription at no extra cost.
    Your Scenario

    Contoso Ltd is onboarding a new junior administrator. You need to create their account in Entra ID, give them read-only access to the lab resource group, and ensure the account is protected by requiring MFA. You also need to review the company's Conditional Access policies to understand what access controls are in place.

    1
    Authentication vs Authorisation

    Before diving into the portal, understand the core distinction that underpins all of identity and access management.

    ConceptQuestion it answersAzure serviceExample
    Authentication (AuthN)Who are you?Microsoft Entra IDUsername + password + MFA
    Authorisation (AuthZ)What can you do?Azure RBACReader role on a resource group
    i Authentication always happens before authorisation. You must prove who you are before Azure decides what you are allowed to do. This distinction appears in every AZ-900 and AZ-500 exam.

    Part 1 - Microsoft Entra ID & Users

    Microsoft Entra ID (formerly Azure Active Directory) is the identity and access management service for Azure. Every Azure subscription is backed by an Entra ID tenant - your organisation's dedicated identity store in the cloud.

    TASK 1.1Explore your Entra ID tenant
    1. In the Azure portal, search for Microsoft Entra ID and click the result
    2. On the Overview page, note your Tenant ID and Primary domain (e.g. yourdomain.onmicrosoft.com) - write these down
    3. In the left menu under Manage, click Users - you will see your own account listed
    4. Click Groups - explore any groups that exist (your free account will have few or none)
    5. Click Licenses - note what Entra ID licence tier you currently have (Free by default)
    REFERENCE - Microsoft Entra ID Overview
    The Entra ID overview - your Tenant ID and primary domain are shown here. Note the left menu for navigating users, groups, and licences. Source: Microsoft Learn
    i Your Tenant ID is a globally unique identifier for your organisation's Entra ID instance. You will use it when configuring app registrations and service principals in AZ-500.
    TASK 1.2Create a new user
    1. In Entra ID, click Users then + New user then Create new user
    2. Set User principal name to jadmin (it will append your tenant domain automatically)
    3. Set Display name to Junior Admin
    4. Under Password, note the auto-generated password or set your own - write it down
    5. Click Next: Properties - optionally add a job title of Junior Administrator and department IT
    6. Click Review + create then Create
    7. Once created, click on the user and note the Object ID - this uniquely identifies the user in Entra ID
    REFERENCE - Create New User Form
    The Create new user form - set the UPN prefix to jadmin and note the auto-generated password before clicking Create. Source: Microsoft Learn
    ! Write down the auto-generated password immediately - you will not be able to retrieve it after leaving this page. The user will be prompted to change it on first sign-in.

    Part 2 - Azure RBAC Role Assignments

    Azure RBAC controls who can do what to which Azure resources. It works by assigning a role to a security principal (user, group, or service) at a particular scope (management group, subscription, resource group, or resource).

    RoleWhat it can doTypical use
    OwnerFull access including managing accessSubscription owners, admins
    ContributorCreate and manage resources, cannot manage accessDevelopers, engineers
    ReaderView resources only, no changesAuditors, junior staff
    User Access AdministratorManage user access onlyIT admins without full owner rights
    i Key exam concept: Role assignments are additive - a user with Reader on a subscription and Contributor on a specific resource group gets both. There is no deny role in standard RBAC (only deny assignments which are rare).
    TASK 2.1Assign the Reader role to your new user
    1. Navigate to your rg-contoso-labs resource group
    2. In the left menu, click Access control (IAM)
    3. Click the Role assignments tab - review any existing assignments
    4. Click + Add then Add role assignment
    5. On the Role tab, search for Reader and select it, then click Next
    6. On the Members tab, click + Select members
    7. Search for Junior Admin and select the user, click Select
    8. Click Review + assign twice to confirm
    REFERENCE - Access Control (IAM) Page
    The Access Control (IAM) page - click Role assignments tab to view existing assignments, then + Add to create a new one. Source: Microsoft Learn
    + The Reader role is now assigned to Junior Admin at the resource group scope. They can view all resources in rg-contoso-labs but cannot create, modify, or delete anything.
    TASK 2.2Verify the role assignment and understand scope inheritance
    1. On the IAM page, click the Role assignments tab - confirm Junior Admin now appears with the Reader role
    2. Now navigate to your Subscription and click Access control (IAM)
    3. Click Role assignments - note that Junior Admin does NOT appear here
    4. This confirms the assignment is scoped to the resource group only, not the whole subscription
    5. Click Check access - search for Junior Admin and review their effective permissions
    i Scope inheritance: Permissions assigned at a parent scope (subscription) flow down to child scopes (resource groups and resources). Permissions assigned at a child scope do NOT flow upward. This is a common AZ-900 exam question.
    4
    Part 3 - Conditional Access & MFA

    Conditional Access is Entra ID's policy engine. It evaluates conditions (who is signing in, from where, on what device, to what application) and enforces controls (require MFA, block access, require compliant device). It is how Zero Trust is implemented in practice.

    ! Conditional Access requires Entra ID P2. Your free account includes a 30-day trial - activate it now from Entra ID - Licences - All products - Try/Buy. It is free and does not require payment details.
    TASK 3.1Explore Conditional Access policies
    1. In Entra ID, scroll down in the left menu to Protection and click Conditional Access
    2. Click + New policy to see the policy creation form - do not save anything
    3. Explore the Conditions section - note the signal types: Users, Cloud apps, Conditions (location, device platform, sign-in risk)
    4. Explore the Access controls - Grant section - note the control options: require MFA, require compliant device, block access
    5. Close without saving - you are exploring the interface only
    6. Click What If in the top menu - this tool simulates what policies would apply to a given sign-in
    7. Set User to your Junior Admin account and click What If - review which policies apply
    REFERENCE - Conditional Access Policy Editor
    The Conditional Access model - signals are evaluated, a decision is made, and access controls are enforced. This is Zero Trust in action. Source: Microsoft Learn
    TASK 3.2Review MFA registration for your account
    1. Navigate to aka.ms/mysecurityinfo in your browser
    2. Review your registered authentication methods - you should have at least one MFA method configured
    3. If not configured, add a method (Microsoft Authenticator app or phone number)
    4. Return to the portal and navigate to Entra ID - Users - Per-user MFA
    5. Note the MFA status for your accounts - this shows which users have MFA enforced
    i MFA vs Conditional Access MFA: Per-user MFA enforces MFA for every sign-in. Conditional Access MFA enforces it only when specific conditions are met (e.g. signing in from outside the office). Conditional Access is more flexible and the recommended approach.

    Part 4 - The Zero Trust Model

    Zero Trust is a security model based on the principle "never trust, always verify". Unlike traditional perimeter-based security (trust everything inside the network), Zero Trust assumes breach and verifies every request explicitly.

    Zero Trust PrincipleWhat it meansHow Azure implements it
    Verify explicitlyAlways authenticate and authorise based on all available data pointsEntra ID, Conditional Access, MFA
    Use least privilege accessLimit user access with just-in-time and just-enough-accessAzure RBAC, Privileged Identity Management
    Assume breachMinimise blast radius, segment access, encrypt end to endNSGs, Private Endpoints, Defender for Cloud
    TASK 4.1Map Zero Trust to what you have built

    Review each Zero Trust principle against the labs completed so far and write your answers in your notes.

    1. Verify explicitly - which controls in this lab enforce explicit verification? (MFA, Conditional Access)
    2. Least privilege - why did we assign Reader rather than Contributor to the Junior Admin? What is the risk of over-permissioning?
    3. Assume breach - how did the NSG in Lab 2 contribute to Zero Trust? What does "minimise blast radius" mean in practice?
    4. Navigate to Entra ID - Security - Identity Secure Score and review your tenant's score and improvement recommendations
    i The Identity Secure Score gives you a percentage score for how well your tenant follows identity security best practices. Each recommendation links directly to AZ-500 concepts.

    Knowledge Check - AZ-900 Exam Style
    Click an answer to check it. One attempt per question. 0 / 5
    1. A company wants to ensure users must use MFA only when signing in from outside the corporate network. Which Azure feature should they configure?
    • A. Per-user MFA enforcement
    • B. Azure RBAC
    • C. Conditional Access policy
    • D. Entra ID Identity Protection
    i Conditional Access policies evaluate signals (including location/network) and apply controls like MFA only when the conditions are met. Per-user MFA forces MFA for every sign-in regardless of location.
    2. A junior analyst needs to view resources in a resource group but must not be able to make changes. Which RBAC role should be assigned?
    • A. Contributor
    • B. Reader
    • C. Owner
    • D. User Access Administrator
    i Reader provides view-only access with no ability to create, modify, or delete resources. This follows the principle of least privilege - granting only the minimum access needed.
    3. Which Zero Trust principle is implemented by assigning the minimum RBAC role required for a task?
    • A. Use least privilege access
    • B. Verify explicitly
    • C. Assume breach
    • D. Defence in depth
    i Least privilege access limits what a user can do, reducing the damage if their account is compromised. Verify explicitly relates to authentication. Assume breach relates to segmentation and minimising blast radius.
    4. What is the correct relationship between Entra ID and Azure RBAC?
    • A. They are the same service with different names
    • B. Azure RBAC handles authentication; Entra ID handles authorisation
    • C. Entra ID handles both authentication and authorisation for Azure resources
    • D. Entra ID handles authentication; Azure RBAC handles authorisation for Azure resources
    i Entra ID verifies identity (authentication). Azure RBAC then determines what that identity is allowed to do with Azure resources (authorisation). They are separate but complementary systems.
    5. A user is assigned the Contributor role at the subscription scope. What is their access to a specific resource group within that subscription?
    • A. No access - subscription roles do not apply to resource groups
    • B. Contributor access - permissions inherit downward from parent scopes
    • C. Reader access only - resource groups are protected by default
    • D. Owner access - subscription contributors get elevated access on child resources
    i RBAC permissions flow downward through the hierarchy. Contributor at subscription scope means Contributor access on all resource groups and resources within that subscription.
