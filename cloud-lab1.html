Lab Overview

In this lab you will deploy a Virtual Machine into a Virtual Network, configure a Network Security Group to control inbound traffic, and connect to your VM through the browser. This covers the core AZ-900 compute and networking objectives and directly lays the groundwork for AZ-500's network security domain.

You have a choice of OS — Windows Server or Ubuntu Linux. The steps are identical up to the connection method: Windows uses RDP, Linux uses SSH. Both are available directly in the browser via the Azure portal.

💡 Not sure which to choose? Pick Windows Server if you're more familiar with Windows environments. Pick Ubuntu if you're heading towards a security or DevOps path — Linux is more common in both.
📋 Your Scenario

Contoso Ltd needs a small test server for a new internal project. Your manager has asked you to provision a VM in the existing resource group, place it on a dedicated virtual network, and ensure it is locked down — only RDP or SSH access should be permitted inbound. All other traffic must be blocked by default. Once deployed, connect to the VM to confirm it's working, then document the NSG rules you've applied.

Part 1 — Create a Virtual Network & Subnet

Before deploying a VM you need a network for it to live on. In Azure, that's a Virtual Network (VNet) with one or more Subnets. Think of the VNet as your private address space in the cloud and the subnet as a logical segment within it.

TASK 1.1Create a Virtual Network
  1. In the Azure portal search bar, search for Virtual networks and click the result
  2. Click + Create
  3. Set Subscription to your free subscription
  4. Set Resource group to rg-contoso-labs
  5. Set Name to vnet-contoso-labs
  6. Set Region to UK South
  7. Click Next: IP Addresses
📸 REFERENCE — Virtual Networks in the Portal
The Virtual Networks overview page — click + Create to start building your VNet. Source: Microsoft Learn
TASK 1.2Configure the address space and subnet
  1. Leave the default IPv4 address space as 10.0.0.0/16 — this gives you 65,536 addresses
  2. Under Subnets, click the default subnet and rename it to snet-contoso-vms
  3. Set the subnet address range to 10.0.1.0/24 — this gives you 256 addresses for VMs
  4. Click Save, then click Review + create, then Create
  5. Wait for deployment to complete — you'll see a notification in the top right
💡 CIDR notation like /16 and /24 defines how many IP addresses are in a range. /24 = 256 addresses, /16 = 65,536. This is AZ-900 exam knowledge — and critical for AZ-500 network segmentation.
📸 REFERENCE — VNet Basics Tab
The Create Virtual Network basics tab — set your name and region here before moving to IP Addresses. Source: Microsoft Learn
📸 REFERENCE — Subnet Address Space Configuration
The subnet configuration — set the name to snet-contoso-vms and address range to 10.0.1.0/24. Source: Microsoft Learn

Part 3 — Create & Configure a Network Security Group

A Network Security Group (NSG) is Azure's built-in firewall for controlling inbound and outbound traffic to resources. It uses rules based on priority, protocol, source/destination IP, and port. NSGs are a foundational security control — and one of the first things covered in AZ-500.

TASK 3.1Create a Network Security Group
  1. Search for Network security groups in the portal and click + Create
  2. Set Resource group to rg-contoso-labs
  3. Set Name to nsg-contoso-vms
  4. Set Region to UK South
  5. Click Review + create, then Create
  6. Once deployed, click Go to resource
📸 REFERENCE — NSG Overview Page
Your NSG overview after creation — the left menu shows Inbound security rules, Outbound security rules, and Subnets. You'll use all three in the next tasks. Source: Microsoft Learn
TASK 3.2Add an inbound security rule

By default, NSGs deny all inbound traffic. You need to explicitly allow the connection port for your OS.

🪟 Windows — Allow RDP
🐧 Linux — Allow SSH
  1. In your NSG, click Inbound security rules in the left menu, then + Add
  2. Set Source to Any
  3. Set Destination to Any
  4. Set Service to RDP — this auto-fills port 3389
  5. Set Action to Allow
  6. Set Priority to 100
  7. Set Name to Allow-RDP-Inbound
  8. Click Add
📸 REFERENCE — Add Inbound Security Rule Panel
The Add inbound security rule panel — set Service to RDP, Action to Allow, Priority to 100. Source: Microsoft Learn
⚠️ Allowing RDP from Any source is fine for this lab but would never be acceptable in production. In a real environment you'd restrict the source to your own IP address or use Azure Bastion. This is a core AZ-500 topic.
  1. In your NSG, click Inbound security rules in the left menu, then + Add
  2. Set Source to Any
  3. Set Destination to Any
  4. Set Service to SSH — this auto-fills port 22
  5. Set Action to Allow
  6. Set Priority to 100
  7. Set Name to Allow-SSH-Inbound
  8. Click Add
📸 REFERENCE — Add Inbound Security Rule Panel
The Add inbound security rule panel — set Service to SSH, Action to Allow, Priority to 100. Source: Microsoft Learn
⚠️ Exposing SSH port 22 to Any source is fine for this lab only. In production you'd restrict to your IP or use a jump host. Expect this to come up in AZ-500.
TASK 3.3Associate the NSG with your subnet
  1. In your NSG, click Subnets in the left menu, then + Associate
  2. Select Virtual network: vnet-contoso-labs
  3. Select Subnet: snet-contoso-vms
  4. Click OK
💡 NSGs can be associated at the subnet level (affecting all resources in the subnet) or at the NIC level (affecting a single VM). Associating at subnet level is more scalable and is the recommended approach.
📸 REFERENCE — NSG Inbound Rules
Your NSG inbound rules should show your Allow rule alongside the default DenyAllInbound rule. Lower priority numbers run first. Source: Microsoft Learn

Part 4 — Connect to Your VM Using Your Own Machine

Now connect to your VM using the native tools on your own computer — this is how it works in the real world. Windows users will use the built-in Remote Desktop client. Mac and Linux users will use a terminal and SSH. Your NSG rule from Part 3 is what makes this possible.

💡 First, grab your VM's Public IP address from the Azure portal — navigate to your VM and it's shown on the Overview page. You'll need it for both connection methods below.
TASK 4.1Get your VM's Public IP address from the portal
  1. In the portal, navigate to your VM
  2. On the Overview page, find the Public IP address field
  3. Copy the IP address — it will look something like 51.140.x.x
  4. Keep this handy — you'll paste it into your RDP client or terminal
📸 REFERENCE — VM Overview showing Public IP
Your VM overview — the Public IP address is highlighted here. Copy this before proceeding. Source: Microsoft Learn
TASK 4.2Connect from your own machine
🪟 Windows — RDP Client
🐧 Linux/Mac — SSH Terminal
💡 Windows has Remote Desktop built in — no extra software needed. Press Win + R, type mstsc and hit Enter to open it.
  1. Press Win + R, type mstsc and press Enter to open Remote Desktop Connection
  2. In the Computer field, paste your VM's Public IP address
  3. Click Connect
  4. When prompted for credentials, enter:
    Username: contosoadmin
    Password: the password you set when creating the VM
  5. Accept the certificate warning — click Yes to connect
  6. Once connected, open Server Manager — confirm the hostname matches vm-contoso-win01
  7. Take a screenshot of the connected desktop as evidence
📸 REFERENCE — Windows Remote Desktop Connection
The mstsc Remote Desktop dialog — paste your VM's Public IP into the Computer field and click Connect. Source: Microsoft Learn
⚠️ If you see "Remote Desktop can't connect" — check your NSG has the Allow-RDP-Inbound rule on port 3389 and that your VM is in a Running state in the portal.
💡 Mac users: open Terminal (Applications → Utilities → Terminal). Linux users: open your terminal application. SSH is built into both.
  1. Open your terminal application
  2. Type the following command, replacing YOUR_VM_IP with your VM's Public IP:
    ssh contosoadmin@YOUR_VM_IP
  3. Press Enter — you'll be asked to confirm the host fingerprint. Type yes and press Enter
  4. Enter your password when prompted (note: characters won't appear as you type — this is normal)
  5. Once connected, run uname -a to confirm the Linux kernel version
  6. Run hostname to confirm the VM name matches vm-contoso-lin01
  7. Run ip addr show to see the private IP assigned within your VNet subnet
  8. Take a screenshot of the terminal session as evidence
📸 REFERENCE — SSH Connection in Terminal
Your terminal should show a successful SSH login with the Ubuntu welcome banner. Run hostname and uname -a to confirm. Source: Microsoft Learn
⚠️ If you see "Connection refused" — check your NSG has the Allow-SSH-Inbound rule on port 22 and that your VM is in a Running state in the portal.
✅ Successfully connected from your own machine? You've deployed real cloud infrastructure, secured it with an NSG, and proved it works using native OS tools. This is exactly how it works in a real cloud environment.

Part 5 — Scenario Challenge

Independent challenge — no step-by-step instructions. Read the brief and figure it out.

📋 Challenge Brief

The security team at Contoso have reviewed your VM and raised a concern — the NSG currently allows RDP or SSH from any IP address on the internet. They want you to tighten this: update the inbound rule so that only your current public IP address is permitted as the source. Then add a second rule that explicitly denies all other inbound traffic on port 3389 (RDP) or port 22 (SSH) at a lower priority. Document what you changed and why priority order matters in NSG rules.

💡 To find your current public IP, search "what is my IP" in a browser. In the NSG rule, set Source to IP Addresses and paste your IP in the source field.

Knowledge Check — AZ-900 Exam Style
Click an answer to check it. Each question can only be answered once. 0 / 4
1. A company needs to prevent all internet traffic from reaching their Azure VMs except on port 443. Which Azure service should they use?
  • A. Azure Firewall
  • B. Network Security Group
  • C. Azure VPN Gateway
  • D. Azure Load Balancer
💡 NSGs are the correct choice for filtering traffic to and from Azure resources using allow/deny rules based on port, protocol, and IP. Azure Firewall is a more advanced, centralised service covered in AZ-500.
2. An NSG has two inbound rules: Priority 100 allows port 22, Priority 200 denies port 22. What happens to SSH traffic?
  • A. SSH traffic is allowed — the lower priority number wins
  • B. SSH traffic is denied — deny rules always override allow rules
  • C. SSH traffic is blocked — rules with higher numbers take precedence
  • D. An error occurs because two rules target the same port
💡 NSG rules are processed in priority order — lowest number first. Priority 100 is evaluated before Priority 200, so the Allow rule is hit first and SSH traffic is permitted. Rules stop being evaluated once a match is found.
3. Which of the following best describes the purpose of a Subnet in an Azure Virtual Network?
  • A. A subnet provides internet connectivity to Azure VMs
  • B. A subnet is a physical network segment in an Azure datacenter
  • C. A subnet is a logical subdivision of a VNet's IP address space
  • D. A subnet replaces the need for a Network Security Group
💡 A subnet is a logical range of IP addresses within a VNet. It allows you to segment your network, apply different security rules to different groups of resources, and organise resources by function or tier.
4. What is the default inbound behaviour of a newly created Azure Network Security Group with no custom rules?
  • A. All inbound traffic is allowed by default
  • B. Only traffic from the same VNet is allowed
  • C. Only traffic on port 80 and 443 is allowed
  • D. All inbound traffic from the internet is denied by default
💡 A new NSG includes a default rule — DenyAllInbound — at priority 65500 that blocks all inbound internet traffic. Traffic within the VNet and from Azure Load Balancer is allowed by separate default rules at higher priorities.

⚠️ Lab Housekeeping — Do This Before You Close the Browser
🚨 This lab deploys a billable VM. You must stop or deallocate it when you're done or it will continue consuming your free credit.

Complete these steps in order:

  1. Navigate to your VM in the portal
  2. Click Stop in the top menu bar — wait for the status to show Stopped (deallocated)
