    Lab Overview

    Azure Storage is one of the most fundamental services in the platform. Everything you build in Azure eventually touches storage - VMs use it for disks, applications use it for files and blobs, and logs flow into it automatically. This lab covers the two most exam-relevant storage types: Blob Storage for unstructured data and Azure Files for shared file access.

    You'll also explore access tiers - a key cost optimisation concept - and redundancy options, both of which appear heavily in AZ-900 exam questions.

    💡 Storage accounts are very cheap for this lab. LRS blob storage costs fractions of a cent for a few test files. Just remember to delete the resources at the end - the housekeeping section tells you exactly what to remove.
    📋 Your Scenario

    Contoso Ltd needs a cloud storage solution for two use cases: a document archive for infrequently accessed compliance files, and a shared drive that remote team members can map on their laptops. Your job is to set up both - a blob storage account with appropriate tiering for the archive, and an Azure Files share that can be mounted on Windows, Mac, or Linux machines.

    1
    Part 1 - Create a Storage Account

    A storage account is the top-level Azure resource that provides a unique namespace for all your storage data. Every blob, file, queue, and table lives inside a storage account.

    TASK 1.1Create the storage account
    1. In the Azure portal, search for Storage accounts and click + Create
    2. Set Subscription to your free subscription
    3. Set Resource group to rg-contoso-labs
    4. Set Storage account name to stcontosolab followed by 4–6 random numbers (e.g. stcontosolab4821) - names must be globally unique, all lowercase, no hyphens
    5. Set Region to UK South
    6. Set Primary service to Azure Blob Storage or Azure Data Lake Storage Gen 2
    7. Set Performance to Standard
    8. Set Redundancy to Locally-redundant storage (LRS)
    9. Click Review + create, then Create
    📸 REFERENCE - Create Storage Account Basics Tab
    The Create storage account form - note the name must be globally unique. Set redundancy to LRS for this lab. Source: Microsoft Learn
    💡 Why LRS? For this lab we use LRS (3 copies in one datacenter) - it's cheapest. In production you'd typically use GRS or RA-GRS for resilience. The AZ-900 exam tests your understanding of when to use each redundancy option.
    TASK 1.2Understand redundancy options

    Before moving on, take 2 minutes to review the redundancy table below - this is heavily tested in AZ-900.

    OptionCopiesLocationsBest for
    LRS31 datacenter, 1 regionDev/test, low-cost scenarios
    ZRS33 availability zones, 1 regionHigh availability within a region
    GRS62 regions (primary + secondary)Disaster recovery, compliance
    RA-GRS62 regions + read access to secondaryRead-heavy apps needing DR
    ⚠️ Exam tip: GRS replicates data to a secondary region but you can't read from it unless Microsoft initiates a failover. RA-GRS lets you read from the secondary at any time. This distinction is a common exam question.

    Part 2 - Blob Storage & Access Tiers

    Blob storage is Azure's object store - perfect for unstructured data like documents, images, videos, and backups. Blobs live inside containers, which are like folders within your storage account.

    TASK 2.1Create a blob container
    1. Navigate to your new storage account in the portal
    2. In the left menu under Data storage, click Containers
    3. Click + Container
    4. Set the Name to contoso-archive
    5. Set Anonymous access level to Private (no anonymous access) - this is the secure default
    6. Click Create
    📸 REFERENCE - Blob Containers List
    The Containers page - click + Container and set access to Private. Never set blob containers to public access unless the content is genuinely public. Source: Microsoft Learn
    ⚠️ Never use anonymous (public) access unless the content is deliberately public-facing. Misconfigured public blob containers are one of the most common causes of cloud data breaches - this is a core AZ-500 concept.
    TASK 2.2Upload files to the container
    1. Click your new contoso-archive container to open it
    2. Click Upload in the top menu
    3. Browse and select your test files (a .txt and an image work well)
    4. Expand Advanced - note the Access tier option. Leave it as Hot for now
    5. Click Upload
    6. Once uploaded, your files appear in the blob list - click one to view its properties
    📸 REFERENCE - Upload Blob
    The Upload blob blade - expand Advanced to see the access tier option. Source: Microsoft Learn
    TASK 2.3Explore and change access tiers

    Access tiers control the cost balance between storage price and retrieval price. This is a key AZ-900 exam topic and a real-world cost optimisation technique.

    TierStorage costRetrieval costBest for
    HotHighestLowestFrequently accessed data
    CoolLowerHigherInfrequently accessed, kept 30+ days
    ColdLower stillHigher stillRarely accessed, kept 90+ days
    ArchiveLowestHighest + rehydration timeLong-term backup, compliance data
    1. Click one of your uploaded blobs to open its properties
    2. Click Change tier (or find the Access tier field)
    3. Change the tier from Hot to Cool - click Save
    4. Note the tier change is reflected in the blob properties
    5. Try changing to Archive - note the warning that retrieval requires rehydration (which can take hours)
    6. Change back to Hot for now - Archive would prevent downloading in this lab
    💡 Rehydration: Moving a blob out of Archive tier back to Hot or Cool is called rehydration. It can take up to 15 hours - this is an exam question. You set the priority (Standard or High) to trade time against cost.
    TASK 2.4Download and verify a blob
    1. In the container blob list, right-click one of your files
    2. Select Download
    3. Verify the file downloads to your machine and opens correctly
    4. Navigate back to the container - note the URL property of the blob in its properties panel
    5. Copy the URL and try pasting it into a browser - since access is Private, you should get an ResourceNotFound or 403 error. This is correct and expected.
    📸 REFERENCE - Download Blob
    Right-click a blob to download it. The URL being inaccessible in a browser confirms private access is working correctly. Source: Microsoft Learn
    ✅ The 403 or ResourceNotFound error on the blob URL is a good thing - it confirms your container access level is correctly set to private. Publicly accessible blob URLs are a major security risk.

    Part 3 - Azure Files & Drive Mapping

    Azure Files provides fully managed SMB file shares in the cloud. Unlike blob storage, Azure Files can be mounted as a drive letter on Windows or a mount point on Mac and Linux - making it a direct replacement for traditional network drives.

    TASK 3.1Create a file share
    1. In your storage account, find File shares under Data storage in the left menu
    2. Click + File share
    3. Set the Name to contoso-shared
    4. Set Tier to Transaction optimized (cheapest for light use)
    5. Click Review + create, then Create
    6. Once created, click the file share to open it
    7. Click Upload and upload any small test file - confirm it appears in the share
    📸 REFERENCE - Azure File Share Overview
    The file share overview - the Connect button generates the mount script for your OS. Source: Microsoft Learn
    TASK 3.2Map the file share as a drive on your machine

    Azure generates a ready-to-run script for mounting the share. Select your OS below.

    Windows
    macOS
    Linux
    1. In your file share, click Connect in the top menu
    2. Select Windows tab
    3. Choose a drive letter (e.g. Z:)
    4. Under Authentication method, select Storage account key
    5. Click Show script - a PowerShell script is generated
    6. Copy the script
    7. On your Windows machine, open PowerShell as Administrator (Win + X → Windows PowerShell (Admin))
    8. Paste and run the script
    9. Open File Explorer - your new Z: drive should appear
    10. Create a test file in the drive and verify it appears in the portal
    💡 The PowerShell script uses net use under the hood with your storage account key as the password. In production you'd use identity-based auth (Microsoft Entra Kerberos) instead - this is an AZ-500 topic.
    ⚠️ If the script fails with a network error, port 445 may be blocked on your network. Try from a home connection or mobile hotspot.
    1. In your file share, click Connect in the top menu
    2. Select the macOS tab
    3. Under Authentication method, select Storage account key
    4. Click Show script - a shell script is generated
    5. Copy the script
    6. Open Terminal on your Mac (Applications → Utilities → Terminal)
    7. Paste and run the script
    8. Open Finder - the share should appear under Network or Locations
    9. Create a test file and verify it appears in the portal
    ⚠️ If mounting fails, port 445 may be blocked. macOS also requires SMB 3 - this is supported on macOS 10.13+ (High Sierra and later).
    1. In your file share, click Connect in the top menu
    2. Select the Linux tab
    3. Under Authentication method, select Storage account key
    4. Click Show script - a bash script is generated
    5. Copy the script
    6. Open a terminal and run: sudo apt install cifs-utils -y (Ubuntu/Debian) or sudo yum install cifs-utils (RHEL/CentOS)
    7. Run the generated script with sudo
    8. Verify the share is mounted with: df -h | grep cifs
    9. Create a test file and verify it appears in the portal
    ⚠️ Linux requires the cifs-utils package and port 445 open. If on WSL2, the mount may need to be done from Windows PowerShell instead.
    ✅ File share mounted and a file visible in both the portal and your local machine? You've just replaced a traditional network drive with cloud storage - and it cost fractions of a penny.

    Knowledge Check - AZ-900 Exam Style
    Click an answer to check it. One attempt per question. 0 / 5
    1. A company stores compliance documents that must be kept for 7 years but are almost never accessed. Which storage tier minimises cost?
    • A. Hot
    • B. Cool
    • C. Cold
    • D. Archive
    💡 Archive has the lowest storage cost but the highest retrieval cost and requires rehydration before access. It's ideal for data that must be retained but is almost never read - exactly like long-term compliance archives.
    2. Which Azure storage redundancy option provides read access to a secondary region without requiring a failover?
    • A. LRS
    • B. ZRS
    • C. GRS
    • D. RA-GRS
    💡 RA-GRS (Read-Access Geo-Redundant Storage) replicates to a secondary region and allows read access at all times. GRS also replicates to a secondary region but doesn't allow reads unless Microsoft initiates a failover.
    3. What is the main difference between Azure Blob Storage and Azure Files?
    • A. Blob storage supports larger files than Azure Files
    • B. Azure Files can be mounted as a drive letter; Blob storage cannot
    • C. Azure Files is cheaper than Blob storage
    • D. Blob storage requires a VNet; Azure Files does not
    💡 Azure Files uses SMB protocol and can be mounted as a drive on Windows, Mac, and Linux - making it a drop-in replacement for a traditional network drive. Blob storage is accessed via REST API or SDK, not as a mounted drive.
    4. A developer accidentally sets a blob container to public access. What is the immediate security risk?
    • A. Anyone on the internet can read the blobs without authentication
    • B. The blobs are automatically encrypted and cannot be read
    • C. Only users in the same Azure tenant can access the blobs
    • D. The storage account is automatically deleted by Azure
    💡 Public anonymous access means anyone with the blob URL can download the file with no authentication. This is one of the most common cloud misconfigurations leading to data breaches. Always use private access unless content is genuinely meant to be public.
    5. Which tool would you use to automate moving blobs from Hot to Cool tier after 30 days of inactivity?
    • A. Azure Policy
    • B. Azure Advisor
    • C. Lifecycle Management
    • D. Azure Monitor
    💡 Lifecycle Management in Azure Storage lets you define rules that automatically move blobs between tiers or delete them based on age or last access time. This is the correct tool for automated tiering - Azure Policy and Monitor serve different purposes.

    Lab Housekeeping - Clean Up Storage Resources
    🚨 Storage accounts and file shares continue to incur small charges even when idle. Delete the resources below when you're done with the lab.
    1. Navigate to your storage account in the portal
    2. If you have a mapped drive, disconnect it first - Windows: right-click the drive in Explorer → Disconnect. Mac/Linux: umount /path/to/mount
    3. Delete the file share: go to File shares → select contoso-shared → click the … menu → Delete
    4. Delete the blob container: go to Containers → select contoso-archive → click the … menu → Delete
    5. Optionally delete the entire storage account: go to the storage account overview → Delete at the top
    💡 If you plan to keep the storage account for the scenario challenge or further practice, that's fine - just delete the file share to avoid ongoing provisioning costs.
